# 极客公园风格写作 Skill

## 账号画像

**极客公园 GeekPark** — 科技创新报道媒体
- 核心定位：发现改变力，支持创新者
- 受众：极客/科技爱好者/创业者/产品经理/工程师
- 调性：**极客精神 + 产品思维 + 创新叙事 + 发现感**
- 代表栏目：极客早知道、行业财报解读、极客周末
- 历史标签：小米、美团、大疆、今日头条的早期报道媒体

---

## 核心写作特征

### 1. 标题公式

```
【发现感型】（极客公园最常用）
→ "第一次深度体验[产品]：它让我看到了[未来的某个可能]"
→ "[公司]正在做一件[大多数人还没注意到]的事"
→ "和[创始人]聊了[时间]，我终于理解了[他们的底层逻辑]"

【产品/技术视角型】
→ "[产品名]的[核心功能]，为什么[与众不同]"
→ "[技术趋势]到了[临界点]，改变[行业]的时候到了"
→ "拆解[产品名]：[N]个值得学习的设计决策"

【创业叙事型】
→ "这家[行业]公司，走了一条[所有人都不看好]的路"
→ "[创始人名]和[产品]的故事：从[起点]到[现在]"
```

### 2. 文章结构

```
【开场：发现感场景】
— 从一次「第一次体验」「初次见面」「现场记录」开始
— 用记者/作者的第一视角带入
— 引出「这件事为什么值得关注」

【产品/技术/公司的「不一样在哪里」】
— 与主流/竞品的差异点是什么？
— 这个不同背后的设计哲学/产品逻辑是什么？
— 极客视角：工程师/设计师会怎么看这个决策？

【创始人/团队的「为什么是这些人」】
— 团队背景与产品选择的匹配度
— 有没有独特的「原点故事」（为什么做这件事）
— 可引用采访/交流内容（但不要通稿化）

【产品/技术的「下一步」】
— 这个方向的下一个里程碑是什么？
— 行业还有哪些人在做类似的事？
— 谁更可能赢，为什么？

【结尾：留给读者的惊喜或启发】
— 用一句话封装「你见过的某种未来的样子」
— 不下终局判断，但给出「值得持续关注的理由」
```

### 3. 写作风格原则

**必须做到**：
- 至少有一处「产品设计决策的深层逻辑拆解」
- 技术描述要专业但不卖弄（工程师觉得准确，非工程师也能懂）
- 体现「发现感」：这家公司/产品为什么值得今天被报道
- 有创业故事或人物故事（不只是功能列表）

**绝对不能**：
- 写成产品说明书或功能列表
- 技术描述错误（会被极客用户挑战）
- 通稿化（完全是公司给的视角）
- 只关注已经成功/知名的公司（要有发现感）

## STEP 流程

### STEP 1：确认参数
```
请告诉我：
1. 报道对象：产品 / 公司 / 技术趋势 / 创始人？
2. 「发现感」在哪里：这件事为什么现在值得被报道？
3. 目标读者：极客用户 / 产品经理 / 普通科技爱好者？
4. 字数：1500 / 2500字
```

### STEP 2：找「不一样在哪里」
在动笔前，确立文章的核心「发现」：
- 这个产品/公司/技术，哪里和别人不一样？
- 这个「不一样」背后的设计/工程或商业逻辑是什么？

### STEP 3：信息采集
重点渠道：
- 产品官网、App Store评论、产品Hunt
- 创始人采访（播客、视频、文字均可）
- GitHub（开源项目）、技术论坛（V2EX、Hacker News）
- 竞品对比分析（Product Hunt、AppFollow）

### STEP 4：按骨架写正文
遵循极客公园「发现感+产品思维+创业叙事」规范

### STEP 5：自检清单
```
□ 开头是否有「发现感场景」（记者视角/第一次体验）？
□ 是否有一处「产品设计决策的深层逻辑拆解」？
□ 技术描述是否专业且准确？
□ 是否有创业故事或人物故事，不只是功能介绍？
□ 结尾是否给出了「值得持续关注的理由」？
□ 是否回避了通稿化视角？
```
